# LOG-035: `_ensure_init` Sets `_first_call_done` Before Initialization Succeeds

**Date**: 2026-04-14
**Type**: CHALLENGE
**Status**: Resolved
**Raised In**: specs/006-ollama-fallback/spec.md — identified during adversarial spec review
**Related ADRs**: ADR-032

---

## Description

`_ensure_init()` in `server.py` sets `_first_call_done = True` at line 75 before `run_sync` (the actual Ollama initialization) completes. If Ollama is down when the first tool call arrives, init fails — but the flag is permanently set. The server never retries initialization, even if Ollama becomes available later in the same session. This blocks any retry-on-recovery story.

## Context

Feature 006 aims to make the memory server resilient to Ollama being unavailable. Part of the recovery story is that if Ollama starts after the server is already running, the server should pick it up on the next tool call. The current initialization design makes this impossible.

## Discussion

### Pass 1 — Initial Analysis

Relevant code in `server.py`:

```python
# lines 73-90 (approximate)
async def _ensure_init():
    global _first_call_done
    if _first_call_done:
        return
    _first_call_done = True        # ← set here, before run_sync
    try:
        run_sync(...)              # ← Ollama call; can fail
    except Exception as e:
        logger.error(...)
        # _first_call_done is already True — no retry possible
```

The flag is set to `True` unconditionally before the try block. On failure, it stays `True`. All subsequent calls skip `_ensure_init` entirely.

### Pass 2 — Critical Review

The spec's Assumptions section (revised) acknowledges this as a known limitation. The question is whether it should be fixed in 006 or deferred.

Arguments for fixing in 006: it's directly related to Ollama-down resilience — if a developer starts Ollama after their first failed recall, they'd expect the next call to work without restarting the server.

Arguments for deferring: fixing it adds scope; the spec's Assumption framing ("known limitation; fixing it is an implementation task") is accurate — it's a contained bug, not a design flaw.

### Pass 3 — Resolution Path

The minimal fix: check whether init actually succeeded before setting the flag.

```python
try:
    run_sync(...)
    _first_call_done = True   # only set on success
except Exception as e:
    logger.error(...)
    # _first_call_done remains False — next call will retry
```

This is a small, low-risk change. The plan phase should include it as an explicit task alongside the `summary_only` restructuring (LOG-034).

## Resolution

Resolved in plan phase. Move `_first_call_done = True` to inside the `try` block, after `run_sync` completes successfully. On failure, the flag stays `False` — the server will retry init on the next tool call. Change is mechanical; no ADR needed.
