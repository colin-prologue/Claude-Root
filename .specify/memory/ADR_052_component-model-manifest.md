# ADR-052: Three-Component Model and `.speckit.json` Manifest

**Date**: 2026-04-18
**Status**: Accepted
**Feature**: 009-speckit-init (Phase 3A prerequisite — absorbed)
**Related ADRs**: ADR-051 (constitution gate), ADR-009 (Python/FastMCP runtime)
**Related LOGs**: LOG-049 (memory coupling)

---

## Context

Feature 009 (speckit-init + upgrade CLI) needs to know which files belong to which component so that `speckit init` knows what to copy and `speckit upgrade` knows what is safe to overwrite vs. what belongs to the project. Without a formal component model, upgrade logic must hard-code file paths and guesses about ownership, which breaks as speckit evolves.

This ADR defines:
1. The three-component model and each component's owned file set
2. File category semantics governing upgrade behavior
3. The `.speckit.json` manifest schema

---

## Decision

### Component Model

Speckit is composed of three independently-deployable components:

| Component | Runtime requirement | Owned files |
|---|---|---|
| **speckit-core** | None | See below |
| **speckit-memory** | Ollama (local) | See below |
| **speckit-oracle** | External oracle repo | None in this repo currently |

`speckit-core` is always installed. `speckit-memory` and `speckit-oracle` are opt-in.

### Owned File Sets

**speckit-core** owns:
- `.claude/commands/` — all skill files (`speckit.*.md`)
- `.claude/agents/` — all agent persona files
- `.claude/rules/` — all rule files (`collaboration.md`, `style.md`, `conventions.md`, `workflow.md`, `agents.md`, `memory-convention.md`)
- `.specify/templates/` — all templates
- `.speckit.json` — manifest (generated by init/upgrade)

**speckit-memory** owns:
- `memory-server/` — entire Python package (pyproject.toml, speckit_memory/, tests/, uv.lock)
- `.mcp.json` — MCP server registration

**speckit-oracle** owns (future):
- TBD — blocked on oracle repo interface contract (Phase 4)

### File Category Semantics

Upgrade behavior is governed by three categories, independent of component:

| Category | Definition | Upgrade behavior |
|---|---|---|
| **System files** | Non-customizable infrastructure: agent defs, templates, skill commands, rules | Overwrite |
| **Config files** | Installation wiring the user may have extended: `.mcp.json`, `CLAUDE.md` | Three-way merge with diff prompt |
| **Project files** | Per-project content: `constitution.md`, all `specs/`, all `.specify/memory/` except `.index/` | Never touch |

The `.specify/memory/.index/` directory is gitignored volatile cache — always regenerated; never in scope for upgrade.

**Agent persona files** (`.claude/agents/`) are System files by default but are a known customization target (Q1 from Open Questions). The upgrade CLI should warn before overwriting them and offer a skip option — see ADR amendment if this changes.

### `.speckit.json` Schema

```json
{
  "speckit_version": "0.1.0",
  "components": ["core", "memory"],
  "installed_at": "2026-04-18T16:00:00Z"
}
```

Fields:
- `speckit_version` — version of the speckit distribution installed (semver string)
- `components` — array of installed component names: `"core"`, `"memory"`, `"oracle"`; `"core"` is always present
- `installed_at` — ISO 8601 timestamp of last `speckit init` or `speckit upgrade` run

The manifest does **not** enumerate owned files — those are derived from the component spec at upgrade time to avoid staleness. An absent or unparseable `.speckit.json` is treated as a pre-speckit project: upgrade offers a migration path rather than failing.

---

## Rationale

**Why not enumerate files in the manifest?** A manifest that lists every owned file would need to be updated every time speckit adds a new skill or template. Deriving owned files from the component definition at upgrade time is more robust — the source of truth moves with the code, not with the installed snapshot.

**Why three categories, not two?** Config files (`.mcp.json`, `CLAUDE.md`) sit between system files (always overwrite) and project files (never touch). They are speckit-owned but the user legitimately extends them (adds MCP servers, adds project sections). A three-way merge preserves both the speckit update and the user's additions.

**Why is the version a single `speckit_version` rather than per-component?** Speckit components are released together from a single repo. Per-component versioning adds complexity without a current use case — the upgrade path is "installed speckit version X, current is Y, diff and apply." If components are ever split into separate packages, this decision is revisited.

---

## Consequences

- `speckit init` writes `.speckit.json` on first install
- `speckit upgrade` reads `speckit_version` to determine migration path; absent = pre-speckit project
- Agent persona files (`.claude/agents/`) need explicit handling in upgrade CLI per Q1 from Open Questions — the upgrade should warn before overwriting
- `speckit-oracle` component scope is TBD; its file ownership is filled in when Phase 4 (010) delivers the export surface design
