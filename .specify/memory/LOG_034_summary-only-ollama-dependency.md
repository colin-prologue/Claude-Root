# LOG-034: `summary_only` Mode Does Not Currently Bypass Ollama

**Date**: 2026-04-14
**Type**: CHALLENGE
**Status**: Resolved
**Raised In**: specs/006-ollama-fallback/spec.md — Assumption #5; identified during adversarial spec review
**Related ADRs**: ADR-032, ADR-037

---

## Description

The spec's P1 user story (US-1) and FR-006 are built on the assumption that `memory_recall` in `summary_only` mode can succeed without contacting Ollama. This assumption is false in the current codebase. Making `summary_only` bypass Ollama requires a code change that is not yet scoped.

## Context

Feature 006 is intended to make the memory server usable when Ollama is unavailable. The strategy for read operations (US-1) is: summary-only recall bypasses embedding (so it works), semantic recall fails with a structured error (so it fails cleanly). This relies on `summary_only=True` being a complete bypass of the Ollama call path.

During spec review, the reviewer read `server.py` and found this assumption is incorrect.

## Discussion

### Pass 1 — Initial Analysis

In `memory_recall` (`server.py`), the call sequence is:

1. Line 120: `_ensure_init()` — calls `run_sync` which calls `embed_fn` (Ollama)
2. Line 123: `_embed_text(query)` — calls Ollama to generate a query vector
3. Line 146: `if summary_only:` — the flag is finally checked here

Both Ollama-dependent calls (lines 120 and 123) execute unconditionally before `summary_only` is ever evaluated. If Ollama is down, the function crashes at line 120 or 123, regardless of `summary_only`.

### Pass 2 — Critical Review

The spec's Assumption #5 noted "must be verified against the current implementation before planning" — but the quality checklist approved the spec while deferring this verification to plan phase. A MUST requirement (FR-006) that depends on an unverified code change is not correctly deferred — it is a hidden prerequisite.

### Pass 3 — Resolution Path

The fix requires restructuring `memory_recall` so that:
1. When `summary_only=True`, skip `_embed_text` entirely (no query vector needed)
2. `_ensure_init()` either: (a) is restructured to skip the embedding setup path when not needed, or (b) is made lazy enough that a `summary_only` call can bypass it

The plan phase must scope this as an explicit task, not treat it as a given.

## Resolution

Resolved in plan phase. `memory_recall` will be restructured to check `summary_only` before calling `_embed_text`. When `summary_only=True`, a new `scan_chunks` function in `index.py` provides non-vector retrieval. See ADR-037 for the design decision.
