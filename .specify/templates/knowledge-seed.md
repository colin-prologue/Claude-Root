# Knowledge Base Seed — Portable Engineering Principles

A starter corpus for a new project's decision knowledge base. These principles were distilled from recorded engineering decisions across multiple prior projects; the originating incident evidence is intentionally omitted, so treat each entry as a strong prior to be confirmed, revised, or retired by local evidence — not as a settled conclusion.

**How to use:** retain these into the project's knowledge base (memory bank, ADR practice, or constitution) as seeded entries. When a local incident confirms one, record that evidence against it. When local experience contradicts one, revise or retire it — every entry lists the conditions under which it should bend. Entries under *Observed patterns* carry weaker evidence than the philosophies and should be easier to overturn.

---

## Philosophies

### Architecture & system design

- **Background services must be stateless and session-independent.** Never tie a daemon's auth or lifecycle to the session that invokes it; use durable, key-based credentials. The session-bound shortcut saves a key but fails as nested auth loops and resource contention that are severe and hard to diagnose.
- **Persistence layers must outlive their consumers.** A storage or memory system whose lifecycle is tied to the process using it accumulates stale state and loses work on interruption. Corollaries: (a) when one datum lands in two stores of asymmetric durability, write the durable store first — the other copy is a derivative, and write order encodes source of truth; (b) halt-state guards (caps, parks, kill-switches, circuit breakers, dedup markers) must persist in the system of record, not process memory — an in-memory guard silently re-grants its full budget on restart, exactly when the process churns.
- **Shared spec, not shared implementation, for cross-dialect duplication.** When the same logical predicate lives in two execution dialects (app code + SQL, client + server validation, ORM + raw query), extract a declarative spec consumed by dialect-specific builders. This dedupes what actually drifts while preserving each path's native optimizations.
- **Protocol uniformity over fast-path latency in audit-trail systems.** Fold fast-path exceptions into the main decision path even at modest latency cost. N decision paths produce N record shapes in the durable log; every fast-path is a future drift risk and cognitive tax on whoever reads the log later.
- **Match defense count to failure modes in deterministic single-caller pipelines.** If the failure mode is an authoring error (not a runtime race) and there is one known caller, static-analysis enforcement suffices; adding runtime enforcement on top is over-design — one of the two defenses is dead weight, and it may introduce its own failure class. Multi-caller or concurrent systems flip this: runtime enforcement becomes load-bearing.
- **Observe metered resources out-of-band.** The component reporting a resource's health must not consume that resource, or the monitor goes dark exactly when the exhaustion it exists to report arrives. Move the observer off the constrained path.
- **Advisory signals must never become control gates.** A signal derived from a stale-prone or eventually-consistent source may tune soft behavior (backoff, alerts) but never a hard gate: behind a gate, a stale value converts a transient condition into a system-wide wedge that outlives the condition — worst when the gated resource is shared, since every consumer wedges at once.
- **Re-baseline after self-authored writes to a monitored channel.** A watchdog that writes to the channel it monitors for external intervention must read back the post-write state and treat that as its new baseline — otherwise its own output triggers it (comment bots keying on "was updated", controllers reacting to their own status patches). Actor filters break under shared identities; timestamp heuristics race.
- **Failure classification is a recovery-policy selector.** Classify failures by what they discredit: a capability wall (permission, dependency, quota) discredits nothing — fix and re-dispatch with full context; a thrash discredits the actor's model — re-dispatch fresh with a facts-only brief; scope overflow discredits the task shape — split, don't retry. Compound failures take the most restrictive applicable policy.
- **Data-shaped extension points over callback-shaped ones in pure cores.** In deterministic cores (reducers, sim steps, transaction appliers), extension points should be an optional out-array/event log the core fills, never a callback it invokes. A callback is an impurity vector held shut only by convention; data cannot run code.
- **Instruments must not change the system they measure.** When a harness/benchmark needs a property the subject lacks (variance, hooks), supply it in the instrument's own layer. When an instrument finding implicates the subject, route it to the subject's design queue as a first-class decision — never patch the subject as a side effect of tooling, never work around defects inside the tool.

### Process, review & delivery

- **Validate safety nets against the actual workflow, not the intended one.** A fallback that doesn't fire in the code paths you actually run offers false assurance; the question is not "does the fallback exist" but "does it fire on the real execution path".
- **Success criteria must name a failure mode.** A criterion that cannot fail under honest evaluation (a trusted reviewer's verbal reaction, a qualitative impression) is a wish, not a gate. Capture qualitative signals as supplementary, never gating.
- **Verify dependency-removal claims against actual artifacts.** "X depended on Y, so Y is gone too" is a narrative until a scan confirms it; most load-bearing for total-absence claims.
- **Preserve manual ritual discipline when automating sequential workflows.** If the human invented a reset between phases, that ritual is evidence the cross-step pollution is a real cost — automate the ritual, don't bypass it.
- **Honest signal over reassuring banner at human checkpoints.** With findings, surface them inline; with no findings, prefer neutral phrasing over an affirmative "✓ all passed" — a green banner converts absence-of-evidence into a green light and accelerates rubber-stamping past the threshold the gate exists to enforce.
- **Manual-use assumption check.** Before specifying the next feature whose predecessor is in real use, check whether hands-on usage has invalidated prior assumptions; review panels cannot replicate the feedback channel of actual use.
- **Vendor-policy preconditions in review gates.** When definition-of-done includes a manual step in a third-party platform (branch protection, billing toggle, marketplace install), review must assert the step is *available on the current plan/region/flag posture*, not merely documented. Paywalled steps are a silent deferral class no architectural or security review catches by default.
- **Format-as-UX needs real-surface review.** Byte-exact fixtures protect against drift, not against the contract being subtly wrong; any human-read output (PR comments, error messages, reports) needs one deliberate look at the real rendered surface before "done".
- **Post-hoc assertion tests for non-invocable systems.** When the system under test can't be invoked programmatically (interactive/conversational/GUI entry points, or prohibitively expensive invocations), structure smoke tests as artifact checkers gated on a prior manual run — an honest checker beats a brittle fake runner.
- **Bilateral contracts cannot be unilaterally frozen.** When two implementations must interoperate byte-for-byte, gate the operationalizing change on (1) a counterparty acknowledgment artifact and (2) pre-computed test vectors both sides can independently reproduce. The contract isn't frozen until two independent producers agree on a canonical example.
- **Cross-cutting invariants hold from the first PR, never deferred to a "hardening" PR.** Scheduling an invariant (auth coverage, cert pinning, no-PII-in-logs) into a late PR creates shippable intermediate states that violate it; per-PR review can't see the violation because it lives in the sequence, not any diff. Enumerate invariants at task-list time and check every PR's scope against them.
- **Triage automated review-bot findings before acting.** Substance + easy fix → implement; substance + architectural implications → ask; style/preference → dismiss with rationale. "Implement everything" rewards bot noise, "ignore the bot" misses real gaps human review missed.
- **Stop at the PR boundary when verification surfaces an earlier-PR defect.** Don't silently fix it in the current PR: escalate with options (inline-fix with explicit scope expansion / accept-as-deferred with a decision record / separate cleanup PR). Silent fixes hide the defect's origin, defeat reviewability, and bypass the owner's decision authority.
- **Medium before capability.** For systems whose value depends on habitual interaction (dashboards, knowledge bases, assistants), the always-reachable low-ceremony access medium is constitutive, not a convenience. Capability built without a medium accumulates unused; reduce the ceremony of reaching the system before scaling what it can do.
- **Edge-case sections are deferral buckets.** Any "edge case" that rewrites cleanly as a functional requirement is a deferred requirement — promote it before the spec closes. The rewrite test is the filter, not wholesale promotion.
- **Snapshot external artifacts before planning.** Spec claims about content living outside the repo (design files, vendor docs, prototypes) are hypotheses until the artifact is pulled in and read; ingestion is the first real scoping pass and belongs pre-plan, never mid-implementation.
- **Escalation calibrated to decision substance, not action category.** Interrupting a human must be justified by contestable content (confidence × blast radius × reversibility), not workflow ceremony. Over-asking is symmetric with over-proceeding: empty prompts train rubber-stamping and degrade the gates that matter.

### Delegation, automation & multi-agent work

- **Decide-async with steelman records, review-bound at the merge gate.** Delegated actors may decide-and-proceed on pivotal judgment calls if each call produces a record steelmanning the rejected option and stating blast radius; review is async during execution but binding at merge. Hard-escalation domains (security, production, secrets, frozen contracts) stay excluded.
- **Verification before autonomy.** No autonomous execution on a task type until independent verification exists for it — otherwise the bottleneck just relocates to the owner as full-time reviewer. Independence means independent *from the author*: verification reusing the author's world-model, fixtures, or self-account certifies the misconception, not the behavior. Corollaries: coordinators trust shared on-disk artifacts, never returned worker summaries; reuse existing independence before adding a dedicated independent step; reviewers read mechanical evidence before the actor's self-report.
- **Workspace isolation before concurrent sessions.** Concurrent agents/sessions get isolated workspaces by default; a shared tree is owned by one primary, and secondaries write only through surfaces designed for concurrency. Shared-tree safety that depends on the accidental shape of the writes is not safety.
- **Enforcement-altitude parity: unenforced intent loses to enforced gates.** Under delegated execution, effective authority belongs to whatever artifact can block progress — a failing test, a rejecting hook, a schema gate — not to whatever is canonically authoritative. Any invariant worth keeping must be encoded at the same enforcement altitude as the artifacts competing with it; prose carries near-zero execution weight. Corollaries: prose-recommended workflow steps never run under delegation — every step gets an artifact + gate or gets deleted; coordination constraints between parallel actors live in machine-checked mechanisms (required CI, up-to-date branch protection, dependency edges), never in prompts or PR bodies.
- **Goal conditions for autonomous loops are adversarial specifications.** A machine-checkable success condition will be satisfied by the cheapest available path. Before a run, enumerate each check's cheapest bypass and price it above the real work — structurally (bypasses impossible or audit-visible), not just in prose.
- **Stale executable artifacts are the dual hazard of delegation-by-artifact.** The property that makes pre-staged artifacts (kickoff tickets, runbooks, goal prompts) reliable — faithful execution without rereading upstream context — makes stale ones dangerous. A model/spec pivot must invalidate or re-point every downstream executable artifact, and the staleness warning must live where the executor actually looks.

### Templates & capability hygiene

- **Templates ship free of author residue.** A reusable scaffold's value degrades with every piece of author-serving content: runtime dependencies that served the author become adoption friction, and dogfood artifacts (the template's own roadmaps, real specs, internal docs) accumulate whenever a scaffold develops itself. Encode the dogfood/template boundary in the filesystem, not convention — adoption friction is set by the worst plausible path users take (cloning), not the supported one.
- **Capability allowlists drift toward over-permissive baselines.** Grants are reactive and visible; removals are invisible. Wide wildcards enter "just this once" and stay forever; auto-grant mechanisms actively undo pruning unless paired with deny rules. Schedule allowlist audits like dependency upgrades. (Applies to CI permissions, IAM, sudoers, extension permissions, agent-tool allowlists.)

### Knowledge capture & curation

- **Prefer conscious capture over automatic retention.** In any memory/knowledge system where retrieval quality matters, automatic high-frequency capture buries signal in noise faster than reflection can surface patterns; explicit capture produces a smaller bank where every entry earned its place.
- **Reduce activation energy by automating extraction, not execution.** Where a habit requires articulating an insight before capturing it, the hardest step is the trigger and gets skipped; invert to auto-extract → human approves/skips.
- **Season commitments: evidence at incident time, principles after recurrence.** Capture evidence immediately; mint standing rules (principles, policies) only after a second incident, second project, or demonstrated retrieval demand. A principle minted at incident+0 is a hypothesis wearing a conclusion's clothes.
- **Cross-project artifacts live in their owning repo, not a consumer's tree.** When an artifact's meaning applies beyond one project (shared convention, schema, design tokens), hosting its file in whichever project surfaced it conflates it with that project's local rules. The owning repo is canonical; consumers reference.

## Observed patterns

Patterns seen in practice one or two times; weaker evidence than the philosophies above, and cheaper to overturn.

- **Decision posture.** Identify hard constraints before exploring options, and record the constraint that ruled alternatives out — it's the revisit trigger. Distinguish pressure-driven from deliberate decisions; don't let pressure choices calcify into norms. Build only what the task requires; a hypothetical second use case doesn't justify an abstraction. Prefer self-contained tooling over external service dependencies when otherwise equivalent. High-confidence claims require citable confirmation points; prefer fewer high-quality signals over comprehensive-but-noisy ones.
- **Aspirational convention drift.** Convention docs describing required behavior in downstream consumers can be complete and accurate while zero call sites exist; tests can pass for the wrong reason (keyword presence). Audit what the convention *says should exist*, or mark it aspirational until wired.
- **Task-tracking state doesn't auto-reconcile across context boundaries.** Completed work can persist as "in progress" across session resets, risking duplicate work or false incomplete reports. Treat status displays as hypotheses; verify against on-disk artifacts.
- **Category-error bugs come in clusters; inventory before declaring done.** A category bug is a missing constraint the whole system tacitly violates — the first instance found is the loudest, not the only one. Spend a fixed budget scanning for cousins, including in *planned-but-unbuilt* work: one review finding traced through a plan can prevent defects before any code exists.
- **Test artifacts must be locally detectable.** Planted issues requiring multi-hop traversal are below any reviewer's detection horizon regardless of rigor; benchmarks built on them measure fixture design failure, not reviewer capability.
- **Audits reliably surface scope creep hidden in fix-prefixed commits.** Two authoring intents in one diff always get seen by the bookkeeping eventually; split or rename up front.
- **Clarification options can collapse orthogonal axes into a false choice.** If a respondent says "I like A but part of C," that signals axis collapse, not indecision — separate the axes and ask each independently.
- **Pre-dispatch state snapshot for delegated-action auditing.** Snapshot observable state (commit hash, filesystem digest, API cursor) immediately before delegating; diff after. Brackets the delegation with independently verifiable state, eliminating reliance on the delegate's self-report.
- **A wrong numeric premise propagates through gated artifacts and survives review.** Panels verify derivations *from* premises, not premises themselves — purging a single false constant can take multiple passes (raw number, derived rates, prose restatements, case variants). Any consistency pass needs an explicit premise-vs-source-of-truth check.
- **Decision-record review verdicts double as autonomy instrumentation.** Ratified-vs-overturned history per actor per task type is exactly the track record an earned-autonomy ratchet needs; one mechanism yields provenance and the trust dataset.
- **Canonical vs. operative artifact split.** When a runtime loads from a deploy location distinct from the version-controlled source, review validates files the runtime never reads, and staleness emits no signal. Symlink or deploy-on-merge; never hand-copy.
- **Test doubles and test suites inherit their author's blind spots.** A fake written by the implementer encodes the implementer's misconception of the external system; author-written scenarios issue only well-formed inputs. Observed in two unrelated domains: large green suites around blocker-severity defects, closed only by verification bringing independent knowledge of the real system.
- **Green-in-isolation PRs can merge red.** Semantic conflicts (one PR deletes what a parallel PR's tests assert on) are invisible to textual merge; prose sequencing binds no one. Required-CI-on-merge-state plus up-to-date branch protection closes the class.
- **Derivative authority may live on mutable surfaces.** A document whose load-bearing claims are all enforced elsewhere at reviewed altitude may live on a low-friction mutable surface (an issue body); only original authority requires the reviewed surface.
- **A bare failure→ready transition is a bug.** Returning failed work to the dispatchable state without an attached diagnosis re-runs the same failure. Make the diagnostic a required state so the naive bounce is structurally unrepresentable.
- **Reconcile a ratified model against running code; route findings by live-break.** After ratifying a target design, audit it against what's actually running; triage each contradiction with one test — does the gap break something live today? Live → patch now; latent → ticket with an explicit reconciliation note, so implementers build against reality, not the whiteboard.
